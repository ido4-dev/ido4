# ido4 — Mission & North Star

## The Problem

AI coding agents (Claude Code, Copilot, Devin, Cursor) can execute development tasks autonomously. But execution without governance is chaos:

- Multiple agents making conflicting decisions with no shared understanding
- No enforcement of development methodology — agents skip steps, ignore dependencies, bypass quality gates
- No audit trail — when something breaks in production, nobody can trace what happened
- No institutional memory — each agent session starts fresh with zero context from previous work

Traditional project management tools (Linear, Jira, Notion) track work after the fact. They don't govern work in real time. They don't enforce rules deterministically. They don't coordinate multiple AI agents. They weren't built for a world where your development team includes autonomous AI.

## The Mission

**Make AI agents more effective, aligned, and context-aware in software development.**

ido4 is not a project management tool. It's the governance layer that gives AI agents shared understanding and enables long-term coherence. GitHub issues aren't tickets — they're **living specifications**: context carriers encoding intent, acceptance criteria, dependencies, and position in the larger architecture.

## North Star Vision

Multiple AI agents working in parallel — each understanding the full picture of what's being built, how their task connects to others, what "done" means, and how to verify it.

The system carries the knowledge, not the agent. AI agents are stateless — each session starts fresh. A fresh agent with the right 100K tokens of context will outperform a "specialized" agent with wrong context. ido4's role is to assemble the right context at the right moment.

## What ido4 Is

- **A governance engine** — deterministic business rules that enforce methodology compliance
- **An institutional memory** — audit trails, context comments, and accumulated project knowledge
- **A context delivery system** — assembling the right information for each agent at each moment
- **A coordination platform** — multi-agent work distribution, task locking, handoff
- **Methodology-agnostic** — profiles define methodologies (Hydro, Scrum, Shape Up), the engine enforces them

## What ido4 Is Not

- Not a project management UI (use GitHub Projects for visualization)
- Not an AI coding agent (it governs agents, it doesn't replace them)
- Not a replacement for Linear/Jira (it's the governance layer underneath, not the tracking layer above)
- Not command-and-control (it empowers agents with context and boundaries, not micromanagement)

## The Consultancy 2.0 Model

ido4 enables a new operating model:

**2 senior humans (PM + Tech Architect) + AI agents + ido4 governance = the output of a 10-person team at a fraction of the cost.**

- The **PM** provides product vision, stakeholder management, and strategic decisions
- The **Tech Architect** provides system design, code quality oversight, and technical judgment
- **AI agents** provide execution capacity — coding, testing, documentation
- **ido4** provides the **governance layer** — deterministic business rules that ensure methodology compliance, audit trails, and quality gates that no agent can bypass

The governance layer IS the scaling foundation. Without it, adding more agents just multiplies chaos.

## Target Audience

**Primary**: Consulting firms and development agencies adopting AI-augmented delivery models — small senior teams that need to govern AI agent output at enterprise scale.

**Secondary**: Enterprise engineering teams deploying multiple AI coding agents who need methodology enforcement and compliance governance.

**Tertiary**: Individual senior developers using Claude Code who want structured project management integrated into their workflow.

## Design Bar

This system is designed for the largest consultancy firms and software service companies that build at scale. The hierarchy, traceability, and context delivery must be architecturally sound and visually compelling. Every number must be real. Every claim must be traceable to implementation. Every governance decision must be auditable.
