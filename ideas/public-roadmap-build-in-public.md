# Public Roadmap & Build in Public

A public-facing roadmap on the ido4 website that serves two audiences at two layers of depth.

**Layer 1 — Curated themes (website).** 4-6 strategic directions framed for enterprise prospects. No implementation details, no dates. Status badges: Exploring / In Progress / Shipped. Answers the question a VP of Engineering asks: "where is this platform going?"

Candidate themes (derived from current ideas backlog):
- **Adaptive Governance** — the platform learns from team patterns and recommends adjustments (governance-maturity-levels)
- **Enterprise Integrations** — connect existing tools as quality signal providers (signal-provider-abstraction, pluggable-quality-gates, external-auditors)
- **Multi-Agent Intelligence** — stuck detection, per-agent permissions, tailored context per role (openhands-patterns)
- **Institutional Memory** — structured knowledge transfer between agents and sessions (context-comment-schema, execution-prompt)

**Layer 2 — Build in public (linked from themes).** The deeper layer for developers and community. Actual specs, decision logs, "here's why we chose X over Y." Each theme links to its build-in-public page showing the real work.

**The meta-story:** ido4 is built using ido4. Specs shaped with ido4shape, tasks governed by ido4 MCP, work distributed across agents. The roadmap isn't just a feature list — it's a live demo. Every shipped item proves the platform works.

**Timing:** Start with Layer 1 (curated themes) on the website now — useful for first client conversations. Launch Layer 2 (full build-in-public) once there are 2-3 shipped milestones to anchor it. An open roadmap with mostly "Exploring" items feels empty; one with shipped items plus what's next tells a momentum story.

**Why not just a GitHub project board:** Enterprise prospects won't dig into GitHub. The website themes layer is the translation layer between internal work and external narrative. Developers who want the raw signal can follow the GitHub.
